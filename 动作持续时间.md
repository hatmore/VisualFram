持续时间是通过 FramImageDuration（动作持续时间）插件中的 ROIActionTracker 类来计算的。以下是详细的计算逻辑：

  1. 数据结构定义

  在 ROIActionTracker.h:9-19 中定义了持续时间记录结构：

  struct DurationRecord {
      uint64_t start_time;  // 动作开始时间戳(ms)
      uint64_t end_time;    // 动作结束时间戳(ms)
      uint64_t duration() const {
          return end_time - start_time;  // 持续时间 = 结束 - 开始
      }
  };

  2. 状态跟踪

  每个动作维护一个状态（ROIActionTracker.h:61-66）：

  struct ActionState {
      bool is_active = false;      // 动作是否正在进行
      uint64_t start_time = 0;     // 开始时间戳
      int miss_count = 0;          // 连续未检测到的帧数
  };

  3. 核心计算流程

  在 ProcessFrame 方法中（ROIActionTracker.h:178-209）：

  1. 检测到目标开始 → 记录 start_time
  if (!state.is_active && anyDetectedInLastThreeFrames[roi.orderId]) {
      state = { true, timestamp, 0 };  // 记录开始时间
  }
  2. 目标持续存在 → 重置 miss_count = 0
  3. 目标消失 → 增加 miss_count
    - 如果 miss_count >= MaxMissFrames（超过最大容错帧数）
    - 记录结束时间并保存持续时间记录：
  roi_durations[roi.orderId].push_back(DurationRecord{
      state.start_time,
      timestamp  // 结束时间
  });

  4. 计算总持续时间

  在 CalculateDurations 方法中（ROIActionTracker.h:215-249）：

  uint64_t total_duration = 0;
  for (const DurationRecord& dr : records) {
      total_duration += dr.duration();  // 累加所有片段
  }
  ptrOutVec->vmMap[order_id]->timeStamp = total_duration;  // 存储到输出

  关键参数

  - MaxMissFrames: 最大容错帧数（允许短暂丢失检测）
  - VideoFps: 视频帧率（用于时间计算）
  - IouValue: IoU阈值（判断目标是否在ROI内）

  总结

  持续时间计算公式：
  总持续时间 = Σ(每次出现的结束时间 - 开始时间)

  如果动作多次出现，会累加所有片段的持续时间。例如：
  - 第1次出现：100ms ~ 500ms → 持续400ms
  - 第2次出现：800ms ~ 1200ms → 持续400ms
  - 总持续时间 = 800ms

  在 FramImageDuration.cpp:217 中可以看到输出格式：
  "动作 " + orderId + "的持续时间为: " + timeStamp + " ms"


动作完成时机：
三个条件：
  1. state.is_active == true → 动作正在进行中
  2. 当前帧没检测到 → detected_roi_order_ids 中不包含该 orderId
  3. miss_count >= MaxMissFrames → 连续丢失的帧数超过阈值

  当这3个条件都满足时，就认为动作1完成了。



所以我的修改方案中的关键点

  在第3步中，我让你在 state = ActionState{}; 之前添加：

  // 更新已记录的最高orderId
  if (roi.orderId > highestRecordedOrderId) {
      highestRecordedOrderId = roi.orderId;
  }

  这样：
  - ✅ 动作1完成 → highestRecordedOrderId 更新为 1
  - ✅ 下一帧的 nextExpectedOrderId = 1 + 1 = 2
  - ✅ 此时如果同时检测到 2、3，只会记录动作2的持续时间

  ---
  举例说明

  假设有 ROI 1、2、3：

  | 时间     | 检测到   | miss_count(1)      | highestRecordedOrderId | 允许记录                |
  |--------|-------|--------------------|------------------------|---------------------|
  | 100ms  | 1     | 0                  | 0                      | ✅ 1（因为 1 == 0+1）    |
  | 200ms  | 1     | 0                  | 0                      | -                   |
  | 300ms  | 1,2,3 | 0                  | 0                      | ✅ 只记录1（过滤掉2,3）      |
  | 400ms  | 无     | 1                  | 0                      | -                   |
  | 500ms  | 无     | 2 (≥MaxMissFrames) | 1（动作1完成）               | -                   |
  | 600ms  | 2,3   | -                  | 1                      | ✅ 只记录2（因为 2 == 1+1） |
  | 700ms  | 2,3   | -                  | 1                      | ✅ 只记录2              |
  | 800ms  | 无     | miss_count(2)=1    | 1                      | -                   |
  | 900ms  | 无     | miss_count(2)=2    | 2（动作2完成）               | -                   |
  | 1000ms | 3     | -                  | 2                      | ✅ 记录3（因为 3 == 2+1）  |

  ---
  参数 MaxMissFrames 的作用

  这个参数定义了"容错帧数"，用于判断动作是否真正结束：
  - MaxMissFrames = 2 → 连续2帧没检测到才算完成
  - MaxMissFrames = 5 → 连续5帧没检测到才算完成（更宽容）

  这样可以避免因为短暂遮挡或检测失败而误判动作完成。
